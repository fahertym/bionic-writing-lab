# Import Plan

This plan keeps real writing imports controlled.

The goal is to build a structured archive, not a folder dump with nicer routes.

## Order Of Operations

1. Small hand-picked fragments
2. One short essay cluster
3. One pamphlet-scale argument
4. One long-form project
5. Major books and large archives only later

## 1. Small Hand-Picked Fragments

Start with a few fragments that are easy to understand and safe to classify.

Good candidates:

- one argument shard
- one debate reply
- one concept stub
- one AI-assisted note
- one rough observation that clearly points toward a later essay

Import them as `fragment` with `status: imported` or `draft`.

Do not publish them immediately. Use them to test concept links, relationship metadata, search facets, and draft preview behavior.

## 2. One Short Essay Cluster

Pick a small cluster around one concept, such as survival pressure, coercion, property, AI authorship, or religion as infrastructure.

Import only enough to test:

- concept assignment
- relationships from fragments to essays
- search facets
- reading path membership
- public/draft visibility

Move pieces from `imported` to `review` before publishing.

## 3. One Pamphlet-Scale Argument

Choose one argument that is bigger than an essay but smaller than a book.

Use this to test:

- pamphlet exports
- relationship chains
- concept concentration
- public route design
- download quality

Do not use the first pamphlet import to solve every export or design problem.

## 4. One Long-Form Project

Only after fragments, essays, and pamphlets behave well, import one long-form project.

Use a contained project with clear source files. Validate:

- section ordering
- full-publication canonical page
- standalone section pages
- downloads
- relationship metadata
- concept links

Keep it draft or review until the structure is reliable.

## 5. Major Works Later

Large projects such as `Empire Exposed`, `Villain in the Verse`, the `Bionic Trilogy`, and other major archives should wait until the intake path is proven.

Those projects need careful handling:

- stable source folders
- explicit manifest metadata
- clear lifecycle status
- concept assignment passes
- relationship/provenance review
- download/export review
- public framing

Do not import major books just because the importer can handle folders.

## Metadata Discipline

For every import:

- preserve Markdown source
- choose the real publication kind
- default to `imported`
- avoid public exposure until reviewed
- attach concepts only when obvious
- add relationships only when they are meaningful
- use reading paths after there are enough pieces for a reader route

The import pipeline is a loading dock. The archive is built by review.
